인증 방식
JWT 발급 구현 후에는 인증이 필요한 API 요청에 아래 헤더를 사용한다.
Authorization: Bearer {accessToken}
현재까지 구현 흐름 기준:
로컬 회원가입/로그인 성공 시 accessToken, refreshToken 반환
Google OAuth 로그인 성공 시 accessToken, refreshToken 반환
기존 임시 토큰 형식: TODO_ACCESS_TOKEN_{userId}
JWT 구현 후 형식: eyJhbGciOiJIUzI1NiJ9...
Auth API
1. 로컬 회원가입
POST /auth/signup
Content-Type: application/json
Request Body
{
  "email": "test@example.com",
  "password": "password1234",
  "nickname": "tester",
  "profileImageUrl": "https://example.com/profile.png"
}
필드
필드	타입	필수	설명
email	string	O	로그인 이메일
password	string	O	원문 비밀번호. 서버에서 BCrypt 해시 후 저장
nickname	string	O	사용자 닉네임
profileImageUrl	string	X	프로필 이미지 URL

처리 흐름
1. provider=LOCAL, providerAccountId=email 로 Account 중복 확인
2. User 생성
3. password BCrypt 해시
4. Account 생성
5. JWT accessToken / refreshToken 발급
Response
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiJ9..."
}
주요 에러
상태	원인
400	이미 존재하는 이메일
400	요청 값 검증 실패

2. 로컬 로그인
POST /auth/login
Content-Type: application/json
Request Body
{
  "email": "test@example.com",
  "password": "password1234"
}
필드
필드	타입	필수	설명
email	string	O	로그인 이메일
password	string	O	원문 비밀번호

처리 흐름
1. provider=LOCAL, providerAccountId=email 로 Account 조회
2. passwordHash 와 입력 password 비교
3. 일치하면 userId 기준으로 토큰 발급
Response
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiJ9..."
}
주요 에러
상태	원인
400	이메일 또는 비밀번호 불일치
400	요청 값 검증 실패

3. Google OAuth 로그인 시작
GET /auth/oauth/google/authorize
설명
Google OAuth A 방식 로그인 시작 엔드포인트다.
브라우저에서 이 URL로 접근하면 서버가 Google 로그인 페이지로 redirect한다.
Request
없음.
Response
302 Found
Location: https://accounts.google.com/o/oauth2/v2/auth?...
처리 흐름
1. Google OAuth authorize URL 생성
2. client_id, redirect_uri, response_type=code, scope 전달
3. Google 로그인 페이지로 redirect
테스트 URL
http://localhost:8080/auth/oauth/google/authorize
4. Google OAuth Callback
GET /auth/oauth/google/callback?code={authorizationCode}
설명
Google 로그인이 끝난 뒤 Google이 호출하는 callback URL이다.
직접 브라우저에서 호출하는 URL이 아니라, Google Cloud Console에 redirect URI로 등록하는 URL이다.
Google Console Redirect URI
http://localhost:8080/auth/oauth/google/callback
Query Parameters
이름	타입	필수	설명
code	string	O	Google이 발급한 authorization code

처리 흐름
1. authorization code 수신
2. Google token endpoint 호출
3. code -> Google access_token 교환
4. Google userinfo endpoint 호출
5. sub, email, email_verified, name, picture 조회
6. provider=GOOGLE, providerAccountId=sub 로 Account 조회
7. 기존 Account 있으면 userId 기준 토큰 발급
8. 없으면 User 생성 + Account 생성
9. JWT accessToken / refreshToken 반환
Response
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiJ9..."
}
주요 에러
상태	원인
400 / 500	authorization code 재사용
400 / 500	redirect_uri 불일치
400 / 500	Google client secret 불일치
500	Google token 교환 실패
500	DB 저장 실패

5. OAuth 로그인 POST
POST /auth/oauth/login
Content-Type: application/json
설명
프론트가 authorization code를 직접 받아서 백엔드에 넘기는 B 방식 테스트용 엔드포인트다.
현재는 Google A 방식이 주 흐름이므로 필수는 아니다.
Request Body
{
  "provider": "GOOGLE",
  "authorizationCode": "4/0AVMBsJ..."
}
필드
필드	타입	필수	설명
provider	string	O	OAuth provider. 현재 GOOGLE
authorizationCode	string	O	Google authorization code

Response
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiJ9..."
}
6. Google OAuth Ping
GET /auth/oauth/google/ping
설명
GoogleOAuthController가 정상적으로 등록되었는지 확인하는 테스트용 API다.
Response
google oauth controller ok

완료

- TokenIssuePort를 통한 토큰 발급 구조
진행 중 / 다음 단계
- 진짜 JWT 발급 구현
- JWT 검증 필터 구현
- Authorization: Bearer accessToken 인증 처리
- /users/me 형태의 현재 사용자 API 정리
- refreshToken 저장 및 재발급 API
- logout 처리
- GlobalExceptionHandler로 에러 응답 표준화
주의사항
1. Google authorization code는 1회용
callback URL을 새로고침하거나 같은 code를 재사용하면 아래 에러가 발생할 수 있다.
{
  "error": "invalid_grant",
  "error_description": "Bad Request"
}
항상 /auth/oauth/google/authorize부터 다시 시작해야 한다.
2. redirect_uri는 완전히 같아야 함
아래 세 곳이 정확히 일치해야 한다.
1. Google authorize 요청 redirect_uri
2. Google token 요청 redirect_uri
3. Google Cloud Console redirect URI
추천 값:
http://localhost:8080/auth/oauth/google/callback
3. 비밀번호는 원문 저장 금지
로컬 로그인 비밀번호는 반드시 BCrypt 해시로 저장한다.
accounts.password_hash
OAuth 계정은 비밀번호가 없으므로 password_hash = null이다.
4. UserController의 create API
최종 구조에서는 직접 유저 생성 API보다는 /auth/signup으로 회원가입하는 흐름이 자연스럽다.
/api/user/create는 현재 테스트 또는 관리용 API에 가깝다.
